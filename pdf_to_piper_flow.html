DocLens: PDF upload to Piper audio generation

This page traces the actual data path in the current codebase, including the data structures used to store extracted PDF text and the algorithms used for extraction, persistence, AI processing, and Piper playback.

Short Answer

Extracted PDF text is stored per page in IndexedDB, not as one giant document string. The main structure is PageDataRecord in the doclens database, pageData object store, keyed as ${docId}:${pageNumber padded to 6 digits}. 

interface PageDataRecord {
  key: string;
  docId: string;
  pageNumber: number;
  text: string;
  columns: number;
  garbageRatio: number;
  pageAi?: PageAi;
}

Complexity notation used below: P = page count, N = total PDF text items, n = text items on one page, L = text length, R = generated AI result length, C = TTS sentence chunks, M = installed Piper voices, and S = ONNX model size.

List-Based Data Model With TC and SC

In your requested model, one page-indexed list is the source of truth for all page operations. Here P means total number of pages in the document. If you call this N, then N = P for this table. 

type PageSlot = {
  pageNumber: number;
  status: "empty" | "extracting" | "ready" | "ai-running" | "ai-done" | "tts-ready";
  pdfText: string;
  textItems: TextItem[];
  columns: number;
  garbageRatio: number;
  aiResult: string;
  ttsChunks: string[];
  audioState: "idle" | "playing" | "paused" | "ended";
  updatedAt: number;
};

type DocumentPageList = PageSlot[];
Step Operation Data Structure Algorithm / Rule TC SC
1 Create custom data type for all variables PageSlot Define a fixed object shape for each page slot. This is a type-level design step. O(1) O(1)
2 Create/initiate zero_list for total pages DocumentPageList Create a page-indexed array with one slot per page: zero_list[pageNumber - 1]. O(1) if using a sparse reserved array like new Array(P). If physically filling every slot with an object, it is O(P). O(P)
3 Fill extracted PDF data into the list zero_list[i].pdfText, zero_list[i].textItems For each extracted page, write the cleaned text, items, columns, and garbage ratio into that page slot. O(Lp + np) per page, where Lp is page text length and np is page text item count. O(Lp + np) per page
4 Modify page data only through the list zero_list[pageIndex] Any update, such as AI status, AI result, TTS state, or cleanup metadata, mutates/replaces the page slot at its index. O(1) for scalar fields; O(R) when replacing result text of length R. O(1) for scalar fields; O(R) for stored result text.
5 Use the list as source for AI operation zero_list[i].pdfText Read page text from the page slot, build the AI payload, stream result back into the same slot. O(Lp + R) client-side, excluding network/model latency. O(Lp + R)
6 Use the list as source for Piper TTS zero_list[i].aiResult, zero_list[i].ttsChunks Split the AI result into sentence chunks, synthesize each chunk, and update audio state in the same page slot. O(R + A), where A is generated audio sample count. O(C) chunk metadata plus bounded audio buffers during playback.
7 Run document-wide operations zero_list Sequentially scan page slots for export, run-all AI, progress counts, or validation. O(P) for metadata scan; O(P + L) if reading all text. O(1) for counters; O(L) if building a combined output.

The current app persists this idea in IndexedDB as separate PageDataRecord records instead of keeping one permanent in-memory list. Conceptually, pageData is the durable version of this page-indexed list.

Step-by-Step Details

1

PDF selection and validation

Dropzone receives a browser File from drag/drop or file input. It validates MIME/name, size, and empty file conditions, then calls onFile(file).

Data: File Algorithm: validation checks Source: src/components/Dropzone.tsx
TimeO(1). Checks MIME/name, size, and zero-byte status.
SpaceO(1). No file bytes are copied in this component. 
File {
  name: string;
  size: number;
  type: "application/pdf" | string;
}
2

Library save

The dashboard converts the file to an ArrayBuffer, calls createDoc(file, buf), then navigates to /doc/$id. Storage creates a UUID, stores the PDF as a Blob, and stores document metadata separately.

Data: ArrayBuffer -> Blob Store: doclens.blobs[id] Store: doclens.documents[id] Algorithm: UUID + IDB safePut
TimeO(B), where B is PDF byte size, because file.arrayBuffer() reads the file and IDB writes the Blob.
SpaceO(B) transient memory for the ArrayBuffer/Blob, plus O(B) persistent browser storage. 
interface DocRecord {
  id: string;
  fileName: string;
  fileSize: number;
  pages: null;
  pageCount: number;
  createdAt: number;
  lastOpenedAt: number;
  aiResults?: AiResult[];
  aiDoneCount?: number;
}
3

PDF rendering path

PdfViewer loads the stored Blob through getDocBlob(docId) and loadPdfDocument(blob). It renders visible pages with pdf.js canvas rendering and overlays a pdf.js TextLayer for selectable text.

Data: PDFDocumentProxy Data: PageMeta[] Algorithm: lazy rendering via IntersectionObserver
TimeO(P) for initial page metadata, plus render cost for visible pages. Each rendered page is roughly proportional to its PDF drawing/text-layer operations.
SpaceO(P) metadata, with canvas bitmap memory bounded by MAX_RENDERED = 5 visible/nearby pages. 
interface PageMeta {
  pageNumber: number;
  cssWidth: number;
  cssHeight: number;
  scale: number;
}

This rendering path is separate from the stored extracted text. It exists so the user can read, select, copy, translate, or speak visible text directly from the PDF view.

4

Analyze document and extract text

The document page runs extractPdfPages(blob, onPage). pdf.js opens the Blob using a temporary object URL, loads CMaps and standard fonts, then visits pages from 1 to pdf.numPages.

Data: PageExtraction[] Algorithm: sequential page traversal Source: src/lib/pdf.ts
TimeO(P + N log N) overall. Each page is loaded once; sorting dominates per-page extraction.
SpaceO(N + L) for collected page extractions, text items, and extracted text until pages are written to IDB. 
interface PageExtraction {
  pageNumber: number;
  text: string;
  items: TextItem[];
  columns: number;
  garbageRatio: number;
}
5

Text item normalization

Each pdf.js text item is mapped into a compact positional structure. The transform matrix gives X/Y coordinates; width and height are copied from pdf.js.

Data: TextItem[] Algorithm: map pdf.js content.items
TimeO(n) per page, one pass over pdf.js text items.
SpaceO(n) per page for the normalized TextItem[]. 
interface TextItem {
  str: string;
  x: number;
  y: number;
  width: number;
  height: number;
}
6

Column detection and reading order

detectColumns clusters text by X position. It buckets X values into 10 vertical bands and looks for peaks above 8 percent of item count. The result is clamped to 1..3 columns. sortByColumns then groups items by column band and sorts each band top-to-bottom, then left-to-right.

Data: buckets: number[10] Algorithm: heuristic X-axis clustering Algorithm: y-desc/x-asc sort
TimeO(n log n) per page. X sorting and per-column item sorting dominate; bucket scanning is constant.
SpaceO(n) per page for sorted copies and column bands. 
columns = detectColumns(items, pageWidth)
sorted = sortByColumns(items, pageWidth, columns)
7

Raw text assembly and cleanup

Sorted items are concatenated into rawText. A line break is inserted when Y position changes by more than 4 units; otherwise a space is inserted between neighboring text fragments. cleanExtractedText removes Private Use Area glyphs, replacement characters, invalid controls, collapses whitespace, and calculates garbageRatio.

Data: rawText string Data: cleaned text string Algorithm: regex cleanup + ratio calculation
TimeO(L) per page. Text assembly and regex cleanup scan the text linearly.
SpaceO(L) per page for raw and cleaned strings. 
{ text, garbageRatio } = cleanExtractedText(rawText)

If garbageRatio > 0.3, the code warns that the PDF may use legacy or non-Unicode fonts, so extraction can be incomplete.

8

Persist extracted pages

The route collects page summaries during extraction and calls writePages(id, collected). This deletes previous page records for the document and writes one PageDataRecord per page into pageData.

Store: doclens.pageData Key: docId:000001 Algorithm: per-doc write lock + cursor delete + put loop
TimeO(P + L). Existing page records are cursor-deleted, then every page text record is written.
SpaceO(L) persistent IDB storage for extracted text; in-memory overhead is the collected page array. 
PageDataRecord {
  key: "abc-doc-id:000001",
  docId: "abc-doc-id",
  pageNumber: 1,
  text: "clean page text...",
  columns: 1,
  garbageRatio: 0
}
9

AI request creation

The right panel loads page text on demand with getPageData(docId, pageNumber). For translation or explanation, buildPagePayload wraps that page text in an OpenRouter chat payload with system and user messages.

Data: Record<string, unknown> Algorithm: prompt template assembly Algorithm: optional previous-page memory excerpt
TimeO(L) for the selected page text and optional memory excerpt copied into the request.
SpaceO(L) for the request strings and messages array. 
{
  model: modelId,
  stream: true,
  temperature,
  max_tokens: 4000,
  messages: [
    { role: "system", content: system },
    { role: "user", content: "--- Page 1 ---\n..." }
  ]
}
10

Stream AI result and store it

streamCompletion streams deltas into a string buffer. UI updates are throttled every 150 ms. When complete, the final string is saved inside the same page record as pageAi.result using upsertPageAi.

Data: PageAi Algorithm: streaming append buffer Algorithm: partial merge into pageData record
TimeO(R) client-side append/persist work, excluding network and model inference latency.
SpaceO(R) for the streamed buffer in memory and O(R) stored as pageAi.result. 
interface PageAi {
  pageNumber: number;
  status: "idle" | "ready" | "running" | "done" | "error";
  result?: string;
  error?: string;
  overrides?: PageOverrides;
  settingsHash?: string;
  updatedAt?: number;
}
11

TTS controller chooses Piper or browser fallback

The TTS button passes state.result to createSmartTtsController. If the engine preference is not browser, the controller lazy-loads the Piper engine, picks an installed voice matching the output language, and calls speakChunked(text, voiceId, signal). If no voice exists in auto mode, it falls back to the Web Speech API.

Data: result string Data: TtsController Algorithm: preferred voice -> language match -> fallback
TimeO(M) to list/filter installed voices and catalog matches, plus lazy import cost the first time Piper is used.
SpaceO(M) for voice metadata in memory; the controller itself is constant size. 
createSmartTtsController(state.result, {
  onState: setTtsState,
  language: eff.language
})
12

Piper model loading

Piper voices are installed from the HuggingFace rhasspy/piper-voices catalog. The app stores each installed model in a separate IndexedDB database, doclens-piper, object store models. A custom LocalVoiceProvider returns the cached ONNX config and a reusable Blob URL to piper-tts-web.

Store: doclens-piper.models[voiceId] Data: ArrayBuffer ONNX + config object Algorithm: lazy engine singleton + Blob URL cache
TimeO(S) to download/install or load an ONNX model from IDB; cached provider lookup is O(1).
SpaceO(S) persistent IDB storage per voice, plus O(S) engine/model memory while loaded. 
{
  onnx: ArrayBuffer,
  config: object,
  installedAt: number,
  language: string,
  langCode: string
}
13

Piper synthesis and playback

speakChunked splits text into sentence chunks with /[^.!?\n]+[.!?\n]*/g. A producer pre-renders chunks through engine.generate(text, voiceId) while the consumer schedules decoded buffers on an AudioContext. The queue is bounded to two decoded buffers.

Data: string[] chunks Data: Float32Array PCM or Blob Data: AudioBuffer[] Algorithm: 2-deep pre-render pipeline
TimeO(R + A) client orchestration, where A is generated audio samples. Neural inference cost is model-dependent and dominates in practice.
SpaceO(C) for chunk strings, plus bounded decoded audio memory of about two chunks because MAX_BUFFERED = 2. 
chunks -> synthesizeAudio -> Float32Array PCM
PCM -> AudioBuffer
AudioBufferSourceNode.start(nextStartTime)
nextStartTime += buffer.duration

Algorithms Summary

Area Algorithm Why it is used
PDF extraction pdf.js getTextContent(), text item mapping, X-axis bucket column detection, top-to-bottom sorting. Reconstructs readable order from positioned PDF glyph/text fragments.
Text cleanup Regex removal of PUA, replacement, invalid control characters; whitespace normalization. Removes glyph garbage from PDFs with incomplete Unicode mappings.
Storage Per-document write mutex, per-page records, cursor range deletion before rewrite. Keeps memory low and avoids race conditions while updating document pages.
AI streaming Append streamed deltas to a buffer, throttle UI flush to 150 ms, persist final text. Maintains responsive UI without writing partial output into IndexedDB constantly.
Piper voice selection Preferred installed voice first, then exact language match, medium quality preference, fallback to first installed voice. Chooses a usable local neural voice with minimal user interaction.
Piper playback Sentence splitting, 2-buffer producer/consumer pre-render queue, sample-accurate AudioContext scheduling. Reduces audible gaps between chunks and bounds memory usage.
